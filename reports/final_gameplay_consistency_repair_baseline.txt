Endurance:
- Main entry: scenes/main/Boot.tscn -> scripts/game_manager.gd.
- Both modes instantiate the same RoadManager and VehicleController.
- start_game() sets road.stage_mode only for STAGE.
- RoadManager.setup() attaches the approved environment wrapper only when stage_mode is true.
- Therefore Endurance falls through to legacy procedural segment visuals for both vehicles.
- VehicleFactory does not select the environment; the defect is the stage_mode visual gate in RoadManager/GameManager.

Audio:
- One AudioManager instance is owned persistently by GameManager.
- Players: Engine, Music and SFX.
- Engine and Music are looped streams started once per game start.
- update_engine() is called every gameplay frame.
- On GRAVEL it calls _play_tone(145 Hz, 0.07 s) whenever SFX is not playing; when the short stream ends it is immediately retriggered.
- The same pattern exists for offroad and drift tones.
- This is a deterministic short-stream pulse train, not a duplicated engine, checkpoint loop or countdown node; it matches the reported continuous tick.

Stage timer/countdown:
- A four-second start countdown already exists and correctly blocks stage_time.
- stage_time is an elapsed stopwatch and is displayed by HUD.update_rally().
- No residual-time value, checkpoint time bonus, duplicate-prevention bonus state or zero-time outcome exists.
- The repair must preserve stage_time for result/penalty compatibility and add one authoritative residual timer, not replace unrelated scoring state.

Landing:
- VehicleController already tracks airborne, air_time and jump_landed.
- GameplayVisualEffectsPilot already listens to jump_landed for the one-shot dust/fragment burst.
- The vehicle visual is a child separated from the CharacterBody3D collider and already receives minor pitch/roll/deep-sand offsets.
- There is no landing compression/rebound state.
- The safe strategy is one visual-only response cycle on the existing visual root; collider, center of mass, velocity, gravity, camera and general handling remain unchanged.
